## 1.2 기술 스택 결정 (PRD 32~47 반영)

| 계층 | 선택 | 근거 |
|------|------|------|
| 프레임워크 | Next.js 14 App Router | SSR + API Routes 일원화, Vercel 무중단 배포 |
| 언어 | TypeScript | 타입 안정성, 의원실 데이터 무결성 |
| 스타일 | TailwindCSS | PRD 명시, 빠른 일관 UI |
| DB/Auth | Supabase | PRD 명시. Postgres + RLS로 권한 분리 |
| AI | OpenAI API | PRD 명시. 서버 측에서만 호출(키 보호) |
| 상태관리 | TanStack Query + Zustand | 서버상태 캐싱 / 가벼운 클라이언트 상태 |
| 폼/검증 | React Hook Form + Zod | API 입출력 스키마 공유 |
| 배포 | Vercel + Supabase | 관리형, 운영 부담 최소 |

## 1.4 AI 비서실장 처리 흐름 (PRD 5장)

```
사용자 자연어 입력
        │
        ▼
[1] 분류 (OpenAI function-calling)
    → category ∈ {민원,조직,일정,홍보,정책,뉴스}
        │
        ▼
[2] 컨텍스트 조회 (Supabase에서 관련 데이터 retrieve)
        │
        ▼
[3] 응답 생성 (4단 고정 포맷)
    핵심요약 / 상황분석 / 추천행동 / 다음할일
        │
        ▼
[4] ai_requests 테이블에 로깅 (감사·통계)
        │
        ▼
화면 카드 렌더 + 바로가기(딥링크) 제공
```

- 분류는 OpenAI **function calling** 또는 structured output으로 강제하여 6개 enum 외 값 차단.
- 응답 포맷은 시스템 프롬프트 + Zod 스키마(`{summary, analysis, recommendation, nextTodos[]}`)로 검증.

## 1.5 권한 모델 (PRD 4장)

| 역할 | 주요 권한 |
|------|-----------|
| 의원 | 전체 조회 + 보고 승인 |
| 정책지원관 | 정책·뉴스·도정질문 전체 CRUD |
| 비서 | 일정·행사·민원 CRUD |
| 홍보담당 | 홍보(보도자료/SNS/문자/웹자보) CRUD |
| 조직담당 | CRM(인물·조직·연락이력) CRUD |

- 구현: Supabase **RLS** + 미들웨어 역할 가드 이중 적용. 상세 정책은 02 문서.

## 1.7 단계별 범위 (PRD 48 MVP → 49 확장)

- **MVP**: 인증/권한, 대시보드, AI 비서실장, CRM, 민원, 일정.
- **2차**: 홍보 생성(보도자료/SNS/문자/웹자보), 정책 지원, 뉴스 모니터링/보고.
